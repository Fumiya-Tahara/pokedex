# tdd.md

## 1. TDDサイクルと基本原則

以下のサイクルを厳格に守り、一歩ずつ着実に実装を進めること。

- **Red → Green → Refactor:** このサイクルをスキップすることは許されない。
- **テストファースト:** プロダクトコードを書く前に、必ず失敗するテスト（Red）を存在させること。
- **最小限の実装:** テストをパスするためだけの「最短距離」のコードを書くこと。過剰な実装は行わない。
- **安定性の維持:** 既存のテストを壊したまま、あるいはスキップしたまま次の工程に進まないこと。

## 2. 実行可能なTODOリスト（Pendingルール）

実装の漏れを防ぎ、設計を明確にするために「Pending」をTODOリストとして活用する。

- **スペックの全列挙:** 実装を開始する前に、期待される振る舞いを `it "does something"` （ブロックなし）の形式ですべてスペックファイルに書き出し、ユーザーに提示すること。
- **1つずつRedにする:** 
  1. 全てが Pending（黄色）の状態で、リストの先頭から1つ選ぶ。
  2. テストブロックを追加し、実行して Red（失敗）を確認する。
  3. 実装を行い Green（成功）にする。
- **完了の定義:** スペックファイル内の全ての Pending が解消され、全て Green になった状態を「機能の完成」とみなす。

## 3. テストコード作成時の厳守事項

### 構成と命名
- `describe`（対象メソッド）、`context`（状況/条件）、`it`（期待値）を明確に使い分ける。
- `context` は `when ...` や `with ...` で始め、状況を具体的に記述する。
- CI時間短縮のため、一つの`context`には一つの`it`を書き、その`it`にテストを集約させる。

### データの扱い
- **FactoryBot:** インスタンス生成は必ず `FactoryBot` を使用し、生のリテラルによる `create` は避ける。
- **DRYの追求:** 共通のセットアップは `let` や `before` を活用する。

### 網羅性と境界値
- **ハッピーパス以外:** バリデーションエラー、例外処理、権限不足などの異常系を必ずテストに含める。
- **エッジケース:** 数値の境界（0, 最小/最大）、空文字、nil、配列の空/多要素などのケースを検討する。

### 外部依存の排除
- **Mock/Stub:** 外部API通信やネットワークを伴う処理は必ずモックする。
- **時間:** 現在時刻に依存する処理は `Timecop` 等を使用して固定すること。

## 4. Claudeへの具体的な行動指示

- 「テストだけ書いて」という指示に対し、実装コードを書いてはならない。
- 「実装して」と言われた際、対応する Red なテストが存在しない場合は、まずテストを書くようユーザーに提案すること。
- **デバッグ方針:** テスト失敗時に原因が不明な場合、推測で修正を繰り返さず、デバッグ用のログ（`pp`など）を出力するコードを一時的に挟み、確実な情報を得てから修正すること。
- **リファクタリングの提案:** Greenになった際、DRY原則や計算量、可読性の観点から改善の余地がある場合はユーザーに提案すること。
